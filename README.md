# Chess Engine

**Live demo:** https://chess-bot-deployment.vercel.app/

A revamped version of an earlier chess bot of mine. The original used a PyTorch neural network as the evaluator, trained on thousands of board positions labeled with Stockfish evaluations. The model itself worked — the training pipeline (in [`train_advanced_model.py`](train_advanced_model.py)) is preserved and the trained weights are still in [`chess_model_improved.pth`](chess_model_improved.pth) — but at 1-ply lookahead the bot played weakly and got stuck in repetition loops. The network knew how to score a position, but not how to plan.

This version replaces the runtime evaluator with a proper **classical search engine**. The ML side of the project is kept in the repo as documentation of the original approach.

## What changed in the revamp

- **Runtime evaluator: PyTorch NN at 1-ply → classical alpha-beta search.** Iterative-deepening negamax with alpha-beta pruning, quiescence search on captures and queen promotions, MVV-LVA capture ordering, killer-move and history heuristics, a Zobrist-keyed transposition table, and a check extension. Plays at roughly amateur club strength.
- **Repetition + 50-move rule baked into the search.** Draws score as 0 inside the search tree, so the engine actively refuses to repeat when it's winning and embraces repetition when it's losing — fixing the move-loop bug the NN had.
- **Tapered evaluation.** Material + piece-square tables + pawn structure (doubled, isolated) + bishop pair + king-shield safety, blended between a midgame and endgame score using a phase weight derived from remaining non-pawn material.
- **Named opening repertoire with UI selector.** Eight curated openings (Italian, Ruy Lopez, Sicilian, French, Caro-Kann, Queen's Gambit, King's Indian, English). Each is compiled at startup from canonical move lines covering both colors; the user picks one from a dropdown and the bot follows the line until either side deviates, at which point the search takes over.
- **Slimmer deploy.** `torch` and `numpy` are no longer pulled into the deployed image — the runtime is pure Python + `python-chess`. Smaller Docker image, faster cold starts on Vercel.

```
engine.py                  # Classical search engine (negamax + α-β + quiescence + TT + opening books)
backend_improved.py        # Flask REST API (used by Render/Docker)
api/index.py               # Same Flask app, packaged as a Vercel serverless function

train_advanced_model.py    # Original PyTorch training pipeline (legacy)
chess_model_improved.pth   # Trained NN weights (legacy)

```

## API

| Method | Path | Body | Returns |
|---|---|---|---|
| `GET`  | `/api/health`           | — | engine status |
| `GET`  | `/api/openings`         | — | list of named openings the bot can play |
| `POST` | `/api/evaluate`         | `{fen}` | white-POV evaluation in pawns + optional `mate_in` |
| `POST` | `/api/best-move`        | `{fen}` | best move (UCI + SAN) |
| `POST` | `/api/suggested-moves`  | `{fen, top_n?}` | top-N candidate moves with scores |
| `POST` | `/api/make-ai-move`     | `{fen, opening?}` | bot's move + resulting FEN |

## Local development

```bash
# Backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
python backend_improved.py           # serves on :5001

# Frontend (separate terminal)
cd chess-frontend
npm install
npm start                             # opens http://localhost:3000
```

## Original ML approach (preserved)

The neural network is a fully connected net with shape **768 → 256 → 128 → 1** and ReLU activations between layers. Each board is encoded as 12 piece-type planes (one per piece, both colors) for 768 binary features, plus hand-crafted positional signals (material balance, mobility, castling rights, center control). Training data was generated by `python-chess` simulating thousands of legal positions through random move sampling, with each position labeled by the Stockfish chess engine's evaluation as the ground-truth target. Optimized with Adam and MSE loss, regularized with dropout and batch normalization.

The model is good at static evaluation but, used at 1-ply, can't compete with an alpha-beta searcher that looks several moves ahead — which is why the runtime evaluator was swapped out.
